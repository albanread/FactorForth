## Why this turned out small

I expected to need significant patching of factor.dll for proper
embedding.  What's actually needed for the foundation is just:

1. A worker thread that owns the libloading'd `Library` (because
   Factor's TLS-resident VM state ties to the calling thread).
2. A command channel for the host to send eval requests.
3. A reply channel for results.
4. A shared `IoState` for the extern functions to find via a
   process-wide `OnceLock<Mutex<Option<Arc<IoState>>>>`.

That's ~300 lines of Rust including comments, error types, and
the per-eval watchdog.  Factor's embedding API was already
sufficient; we just needed to organise the calls into a
threadsafe shape.

## The architectural shape

```
Host main thread                    Worker thread (owns VM)
─────────────────                   ─────────────────────────
session.eval("...")
  ├─ Watchdog::arm
  ├─ cmd_tx.send(Eval { ..., reply })
  └─ reply_rx.recv()  ◄─────┐
                            │      worker_main loop:
                            │        cmd_rx.recv() → Eval { source, reply }
                            │        eval_inner via nf_eval_string
                            │        (Factor may call rt_* extern fns;
                            │         they look up CURRENT via OnceLock)
                            └──────  reply.send(result)

session.feed_input(b"...")
  └─ io_state.input_q.lock().push
     io_state.input_cv.notify_all()
                            ┌──────  rt_read_char on worker
                            │        ├─ lock input_q
                            │        ├─ if empty: input_cv.wait
                            └──────  ◄─ wake on notify, pop, return
```

The main thread never touches the VM directly.  KEY blocking on
empty input doesn't deadlock because the main thread is free to
call `feed_input` independently.

## What IoMode lets us do

```rust
// Tests: pre-feed input, capture output deterministically.
let out = Arc::new(Mutex::new(Vec::new()));
let session = Session::new(SessionOpts::defaults_for_crate(
    IoMode::Test { input: b"5\n".to_vec(), output: out.clone() }
))?;
session.eval(&compile("." what? "  accept .")?)?;
assert_eq!(out.lock().unwrap().as_slice(), b"what? 5 ");

// CLI: route through stdin/stdout.
let session = Session::new(SessionOpts::defaults_for_crate(
    IoMode::Terminal
))?;

// GUI: feed keystrokes from the event loop, paint output into a pane.
let session = Session::new(SessionOpts::defaults_for_crate(
    IoMode::Gui { on_write: Box::new(|ch| pane.append_byte(ch)) }
))?;
// later, on a key event:
session.feed_input(&[key_char]);
```

Same `Session::eval` everywhere.  Same compiled IR.  The host
decides where bytes come from / go to.

## What 3.1b needs to wire

The extern functions exist (`rt_read_char`, `rt_write_char`,
`rt_read_line`) but Factor doesn't know how to call them yet.
Three pieces:

1. **Library registration** in Factor: at session startup the
   worker evals
   ```factor
   USING: alien.libraries ;
   "nf-host" "<path-to-current-exe>" cdecl add-library
   ```
   so Factor's FFI can resolve `nf-host` to GetProcAddress on
   our binary.

2. **Symbol export** on the Rust side: Windows .exe files don't
   export `#[no_mangle] pub extern "C"` symbols by default the
   way `cdylib`s do.  We need either a build.rs emitting
   `cargo:rustc-link-arg=/EXPORT:rt_read_char` per function,
   or a `.def` file, or `#[link_section = ".drectve"]` in code.
   Worth verifying which works before committing to a path.

3. **forth.runtime declarations** + tuple methods: declare the
   three functions via `FUNCTION:`, then define
   `nf-host-output-stream` and `nf-host-input-stream` tuples
   with the Factor stream protocol methods, then a
   `install-host-streams` word the session calls at startup.

After 3.1b lands, the migration in 3.1c is mostly cosmetic.

## What surprised me

How clean the singleton-enforcement check is.  Factor genuinely
is one-VM-per-process; trying to support multiple Sessions would
mean either patching Factor (hard) or accepting deeply weird
cross-VM behaviour (worse).  Just returning
`SessionError::AlreadyRunning` on the second `Session::new` is
both correct and matches what users expect — "you wanted two
Forth REPLs in one host?  Spawn two host processes."
