# 2026-05-24 — M2.10 ANS strings that don't crash

**Shipped**

- `S" text"` now correctly produces (c-addr, u) per ANS spec —
  two cells, not one Factor string.  Backing byte-array is GC'd;
  no PAD, no shared temporary, no clobbering.
- `TYPE` reclaimed for ANS semantics: takes (c-addr, u), writes
  u bytes.  Decodes UTF-8 internally.
- `CMOVE` (forward byte copy), `FILL` (fill bytes with a byte
  value), `BL` (the ASCII space constant, 32).
- `." text"` continues to work via a dedicated `print-string`
  helper — TYPE was reclaimed for the (c-addr, u) form, so `."`
  needed its own path.
- 4 new integration tests, the original 30 still pass.  88 lib +
  34 integration tests overall.

**Open**

- `COUNT` (counted-string to addr+len) — counted-string format
  is rarer than (c-addr, u); add when needed.
- `CMOVE>` (backward copy for overlapping ranges) — same.
- `COMPARE`, `SEARCH`, `-TRAILING`, `BLANK`, etc. — flesh out
  the comparison/search vocabulary as users hit them.
- Number formatting (`<# # #S #> SIGN HOLD`) — its own thicket.
- C" (counted-string literal) — defers to S" for now.

## The user's framing

> *"did we address the dreadful ANS string words yet, dont quote
>  me on that, but my programs always crash when I use them."*

A truthful observation about ANS Forth's string vocabulary: it's
the spec's most crash-prone area, mostly because of three
historical decisions:

1. **PAD as shared temporary.**  Many words (number-formatting,
   `S"`-at-interpretation, some I/O helpers) return strings in
   a single 64-byte scratch buffer.  Two calls in a row → first
   string clobbered.  Programs written assuming the strings
   live "long enough" crash silently on the second call.

2. **Two address conventions.**  c-addr (points to first
   character) vs. counted-string (points to a length byte, then
   characters).  `COUNT` exists to convert; users forget which
   they have and call words for the wrong shape.

3. **CMOVE direction.**  Forward-copy only safe when source <
   destination; backward (`CMOVE>`) needed for overlap the other
   way.  Pick the wrong one and you overwrite source mid-copy.

Our nf-addr model sidesteps all three:

- **No PAD.**  Strings are byte-arrays.  Each S" allocates a
  fresh one.  Factor's GC keeps them alive as long as anyone
  holds a reference; once dropped, they're gone.  No clobbering.
- **One address convention.**  (c-addr, u) only.  c-addr is an
  nf-addr; u is the byte length.  C" / counted-strings stay
  deferred; users who reach for them are old-school ANS folk
  and we can support them later.
- **Forward copy that you can't get wrong.**  CMOVE works on
  separate buffers (the common case).  Overlapping uses get
  CMOVE> when we add it.

## What the IR looks like

For `s" hello" type cr`:

```factor
"hello" forth.runtime:s-quote-runtime forth.runtime:type forth.runtime:cr
```

`s-quote-runtime` materialises the compile-time Factor string into
a runtime (nf-addr, u) pair, allocating a fresh UTF-8-encoded
byte-array per call.  `type` then writes the bytes.

For `." text"`:

```factor
"text" forth.runtime:print-string
```

No round-trip through nf-addr — the user can't observe the
address either way for output-only text.  `print-string` is a
thin Factor `write` wrapper.

## The bugs that surfaced

Two stack-order bugs in the same session, both classic Forth
patterns:

### FILL's then-branch consumed stack-items that weren't there

```factor
::  fill  ( c-addr u char -- )
    u 0 <= [ 2drop drop ] [ ... ] if ;     ← then-branch wrong
```

The `:: locals` binding moved the inputs OFF the stack; the
then-branch `2drop drop` was a leftover from a non-locals draft.
Factor caught it with a stack-effect-inference error:

> The input quotations to 'if' do not all leave the stack at
> the same height

Cleanest of the recurring stack-shuffling bugs because Factor
told me exactly what was wrong.  Fixed to `[ ]`.

### nf-addr+ argument order in FILL/CMOVE

`nf-addr+` is `( addr n -- addr' )`.  I had written:

```factor
ch i c-addr nf-addr+ nf-c!
```

So the stack going into `nf-addr+` was `(ch i c-addr)` — top
two are (i c-addr), but `nf-addr+` wants (addr n) where addr is
below n.  So it tried `off>>` on i (an integer) and crashed
with "Generic word `off>>` does not define a method for the
fixnum class."

This is the SECOND time this kind of bug has hit nf-addr+ in
the same week.  The first was `[ off>> + ] [ ba>> ] bi` applying
its quots to the wrong stack item.  The second was this arg-
order one.  Recurring lesson: the operand-order conventions of
ANS (addr below value), Factor (consumer-on-top), and our
helpers (addr below offset) all subtly disagree, and the only
way to get it right is `:: locals` plus deliberate naming.

Fixed by writing the body deliberately:

```factor
[| i |
    ch  c-addr i nf-addr+  nf-c!     ! ch at c-addr+i
] each
```

The comment is the spec.  Readable, correct.

## What this actually enables

The cbuffer + FILL combination gives users:

```forth
80 cbuffer line
0 line 80 bl fill        \ blank the buffer
                         \ ...write into it byte-by-byte...
0 line 80 type           \ display
```

Which is the classic ANS pattern for line-oriented I/O, but
without PAD's surprises.  Two cbuffers + CMOVE gives you safe
buffer-to-buffer copy.

The Mandelbrot demo doesn't need strings, but the broader
"applications" goal sure does — text editing, file I/O,
hash-table-style key buffers, all need this vocabulary.

## What's still missing

The big one is **number formatting**.  ANS `<# # #S #>` is a
finicky ceremony that uses HOLD as another shared temporary.
I think we ship a clean alternative — `n>$ ( n -- c-addr u )`
that returns a fresh buffer with the number's decimal
representation — and treat the legacy `<# ... #>` as deferred.

`COMPARE`, `SEARCH`, `-TRAILING`, `BLANK` are all easy
additions when a user actually hits them.

`COUNT` for counted-strings is a sub-day add when someone
loads ANS code that uses them.

## Pattern recognition

This is now the THIRD milestone in a row where the implementation
was 50–200 lines of Rust + a few additions to forth.runtime,
plus 4 integration tests.  The architecture has settled.  Each
new ANS surface lands as:

1. Parser entry (if it's a defining word) OR resolve-table entry.
2. Effect-table entry.
3. forth.runtime word.
4. Maybe an emit-side translation.
5. Tests.

The sema spine carries everything else.

## Pipeline state

```
ANS Forth source
  ↓  lex          ✓ S"/C"/." literals; numeric prefixes; etc.
  ↓  parse        ✓ defs, control flow, defining-words, collections
  ↓  sema         ✓ resolve, effect, escape, call graph
  ↓  emit         ✓ synth annotations, peep-emits, collections,
                    strings via s-quote-runtime + print-string
  ↓  eval         ✓ embedded VM round-trip
  → output + warnings
```

88 lib + 34 integration tests.  About 4 seconds of wall-clock
to run the whole integration suite (each test spins up a fresh
VM; 34 × ~120 ms each).
