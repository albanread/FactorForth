# M2.11 — error translation (partial) (2026-05-24)

Goal: turn Factor's stack-trace error rendering into single-line
ANS-style messages so the eventual Forth-2012 test runner (#41) has
readable failures.

## What shipped

### 1. `error-stream` bound to nf-host-output-stream

`install-host-streams` now sets `error-stream set-global` in
addition to `input-stream` and `output-stream`. Previously errors
sometimes landed nowhere visible because Factor's `print-error`
tried to write through an unbound error stream. With this binding,
the stock `print-error` output flows into our captured `Vec<u8>`.

Pre-#35:
```
captured = "The word rt1 cannot be executed because it failed to compile\n\nError in print-error!\n"
```

Post-#35: actual error class names appear (truncated mid-format by
the prettyprint path because nf-host-output-stream doesn't
implement all the streaming features pretty-printing wants, but
the relevant words are visible — "Generic word ...", "no method",
"out of bounds", etc.).

### 2. Session survives errors

The big invariant: a Factor error doesn't kill the worker thread.
After an error, `Session::eval` returns and subsequent calls
succeed. Confirmed by `err_does_not_kill_session` (test: trigger
no-method → re-evaluate `21 21 + .` → assert "42" in output).

### 3. `nf-format-error` / `nf-eval-callback` machinery in place

`forth.runtime` §13 contains the full ANS THROW-code translator
infrastructure:
- `nf-format-error` ( err -- ): dispatches on error class, prints
  one-line ANS-style message
- `nf-format-kernel-error` for the `{ KERNEL-ERROR n ... }` array
  shape (stack underflow, etc.)
- `nf-format-tuple-error` for tuple conditions (division-by-zero,
  no-method, undefined-word, bounds-error)
- `nf-eval-callback-body` wraps `(eval)` in `recover` with
  format-error as recovery
- `install-nf-eval-callback` swaps `OBJ-EVAL-CALLBACK`

**But it's NOT wired in.** See "what's deferred" below.

## What's deferred — the alien-callback boundary problem

The custom eval-callback is built but `install-nf-eval-callback`
isn't called during session setup. Reason:

Wrapping `(eval)` in `recover` works fine when called from inside
another word (which is exactly what Factor's stock `eval>string`
does: `[ ... [ (eval) ] [ print-error ] recover ... ]
with-string-writer`). But putting that recover as the OUTERMOST
quotation of an `alien-callback` reliably trips
`combinators:wrong-values → kernel:die` on certain error paths
(bounds-error from `$slice` reproduces 100% of the time).

Tried:
- `::` locals form: crashes
- Fried-quotation `'[ _ ... ]`: crashes (same way)
- Inner safety-net recover (`[ format ] [ drop ] recover`): crashes
- Removing `with-file-vocabs` and `parser-quiet?`: crashes

The crash trace shows Factor's per-callback data-stack-effect
check (`check-datastack`) running between our body's return and
the C ABI return, and that check decides the actual stack
doesn't match the declared `void* { c-string } cdecl`. Why
specifically on the bounds-error path and not the no-method
path, when both are tuple-throws, is unclear without deeper
spelunking into Factor's continuation + alien-callback machinery.

Filed as **#47**. Until cracked, we use Factor's stock
eval-callback (which has its own recover via `eval>string` —
that's INSIDE the callback's named-word call chain, where recover
works fine).

## Other deferred: hardware-trap errors

Integer divide-by-zero (`5 0 /`) and stack underflow (`drop` on
empty stack) crash the process at the OS level regardless of
`params_set_signals` setting:

| signals | DBZ outcome | underflow outcome |
|---|---|---|
| 0 (off) | STATUS_INTEGER_DIVIDE_BY_ZERO | reads off-the-end memory |
| 1 (on)  | STATUS_INTEGER_DIVIDE_BY_ZERO  | STATUS_ACCESS_VIOLATION |

Factor's signal-to-condition translation works in standalone
factor.exe (which has its own SEH installation in `vm/factor.cpp`)
but isn't fully wired up in our embedded `nf_init_factor` path.

Filed as **#48**. Probably needs SEH config in vm/factor.cpp, OR
software-checked wrappers around `/`, `MOD`, etc. that throw
catchable conditions on zero divisor.
