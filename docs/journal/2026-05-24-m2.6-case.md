CASE was structurally simple after IF/THEN was in place.  The
parse is a wrinkle on the block-until-terminator pattern: we
collect expressions until we see `OF` (then they're the match
expression for an upcoming arm) or `ENDCASE` (then they're the
default branch).  Nested CASEs are absorbed by `expr_one`'s
recursion the same way nested IFs are.

Emit is a recursive nested IF chain:

```forth
n CASE
  1 OF body1 ENDOF
  2 OF body2 ENDOF
  default
ENDCASE
```

becomes

```factor
dup 1 = [ drop body1 ] [
  dup 2 = [ drop body2 ] [
    default kernel:drop
  ] kernel:if
] kernel:if
```

The dispatch value rides on the stack through each `dup … =`
comparison; `drop body_i` on a match consumes it; the
innermost else (`default kernel:drop`) handles the
no-match-no-default case by dropping the dispatch value
straight through (ENDCASE's contract).

Vacuous `CASE ENDCASE` emits just `kernel:drop`, which is the
ANS-prescribed single-drop behaviour for an empty CASE.

## What actually bit me: emit-time vocabs

The CASE machinery surfaced a latent bug in `vocabs_needed()`.

`resolve.rs` walks the AST and records every ANS word's mapped
Factor target.  `vocabs_needed()` then unions those targets'
vocabs into the IR's `USING:` line.  Sound design, except the
emitter doesn't only output words that resolve saw — it also
outputs literals:

- `."` strings emit as `"text" forth.runtime:type`
- IF emits `kernel:if` and `kernel:when`
- BEGIN/WHILE/REPEAT emits `math:zero?` and `kernel:loop`
- CASE emits `kernel:if`, `kernel:drop`

These are *emit-time fixed strings*.  Resolve doesn't see them
because they're not in the input AST.  Until M2.6 the symptom
hid because most test programs used user-facing ANS words like
`.` or `+` that brought `forth.runtime` and `kernel` in
naturally.  The first CASE test that didn't reach any such word
— `case 1 of ." one" endof endcase` with input 7 — produced an
IR that referenced `forth.runtime:type` without `forth.runtime`
in `USING:`.  Factor parser: "No word named 'forth.runtime:type'
found in current vocabulary search path."

Confusing because the word *did* exist in the loaded image.
Factor's FQ resolution still requires the vocab to be in the
parser's search path.  USING: is what tells the parser where to
look.

Fix: hardcode the four emit-time vocabs (`kernel`, `math`,
`io`, `forth.runtime`) as always-present in `vocabs_needed`.
They're small to include even when unused — Factor's USING:
mechanism is just adding to the parser's lookup set, not
loading additional code.

The cleaner long-term answer is to track emit-time vocab uses
via the emitter itself — same way resolve tracks ANS-word
vocabs.  Refactor for M2.7 if we end up with more emit-time
vocabs.  For now: hardcoded.

## What surprised me

Nothing dramatic this time.  The save-image and inline lessons
from M2.5 carried over for free — no new surprises in the
runtime side, no flagged behaviour deviations.

CASE was, on net, the easiest milestone to ship since the
front-end pieces started landing.  That tracks: the
structural complexity is paid up front in M2.4's `parse_block_until`
helper and the `expr_one` recursion pattern; once those exist,
every new control structure is an incremental specialisation.

If the rest of Phase 2 lands at the same pace, we'll be in
Phase 3 (the GUI binary + iGui plumbing) before the week is out.
