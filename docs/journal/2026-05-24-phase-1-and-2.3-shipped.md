## How we got here

Yesterday we'd shipped the patched `factor.dll` with the embedding
API (`nf_init_factor`, `nf_eval_string`, `nf_run_startup`, etc.) and
a C smoke test that proved end-to-end embedding worked.  The
question on the table: how much of the ANS Forth front end belongs
in Rust, and how much in Factor?

We'd been drifting toward Option B' — Rust does the *entire* front
end (lex, parse, parsing words, stack-effect inference, dictionary,
error model); Factor sees only canonical machine-generated source
as an internal IR.  The user writes ANS Forth.  The user never sees
Factor — not in source, not in errors, not in tooling.

The user reset the conversation cleanly: "Ok enough sentiment,
lets build!"  So we did.

### Phase 1: the two vocabs

The Factor side of the split is small by design.  Just two vocabs:

- **`forth.runtime`** (339 lines) — words ANS Forth needs that
  Factor doesn't directly express, or expresses with different
  semantics.  The `nf-addr` memory model (byte-array + offset),
  return stack via `fstack`, do-loop frames, ANS-style I/O, ANS
  booleans (-1/0), floored mod, FP aliases.
- **`forth.wf64-gfx`** (188 lines) — FFI declarations for the
  host's `rt_gpane_*` extern functions.

The `nf-addr` model is the load-bearing piece.  ANS addresses are
"pointers into a byte-addressed memory" but Factor has no such
memory — its world is tuples and GC'd objects.  We model an
address as `T{ nf-addr { ba byte-array } { off integer } }` and
implement `@`/`!`/`c@`/`c!` via Factor's `alien-accessors`
primitives operating on the byte-array's pinned data pointer.

Byte-arrays are pinned by Factor's GC, so the pointer is stable
across calls.  Allocation through `<variable>` and `<buffer>` just
makes a fresh byte-array and wraps it.

**First real bug, found by running compiled code**: `@` and `nf-!`
had the wrong `bi` order.  I'd written

```factor
: nf-! ( n addr -- )
    [ off>> ] [ ba>> ] bi set-alien-unsigned-cell ; inline
```

which produces `( n off ba )` on top of the stack.  But the
primitive wants `( value c-ptr n -- )`.  Off and ba in the wrong
order.  Fix: swap the quotations to `[ ba>> ] [ off>> ] bi`.

Lesson: until you run the code, you don't know if your stack
shuffling is right.  Hand-tracing only catches the bugs you
already suspect.

### The image-build problem

We bootstrap from the stock `factor.image` (built by the unpatched
`factor.exe`), add `E:/NewFactor/factor` as a vocab root, USE: both
vocabs, `init-remote-control`, `set-startup-quot [...]`, and
`save-image-and-exit` to `images/nf-mandelbrot.image`.

The startup quotation is the wiring that makes `nf_eval_string`
work: `init-remote-control` installs `OBJ_EVAL_CALLBACK` so the
embedding API has a callback to dispatch to.  Without it, every
`nf_eval_string` call segfaults.

The `nf_run_startup` step is what runs that quotation.  I missed
this on the first cut — I'd structured the Rust embedding as
init→eval and the first eval crashed.  Re-reading `smoke6.c` from
the earlier C smoke tests turned up the missing `run_startup`
call between init and eval.  Documented in both `embed_smoke.rs`
and `tests/smoke_runtime.rs` so it doesn't get re-discovered.

The image weighs in at 134 MB — that's the full bootstrapped
Factor stdlib carried into the image.  A slim image (boot-image +
just our two vocabs) would be ~30 MB but that's a future
optimisation.

### Phase 2: the front end

M2.1 — **tokeniser**.  Pure UTF-8 byte-stream walker.  ANS Forth
is whitespace-delimited with a handful of parsing-word prefixes:
`\` (line comment), `(` (block comment), `."`/`S"`/`C"` (strings).
Numbers detected up front by shape: `$ff`, `%1010`, `#42`, `0xCAFE`,
`-7`, `1.5e3`.  Every token carries a `Span` with 1-based line/col
+ byte offset.

One sharp edge I tripped on: `0xE83800` was getting parsed as a
float because my `looks_like_float` heuristic matched any digit-
shaped token containing `E`.  Fix: bail out of float detection
when the token has a non-decimal base prefix.  ANS hex with
embedded E is integer, not exponent.

M2.2 — **parser**.  Recursive-descent over the token slice.
`Program → Item → (Definition | TopLevel) → Expr`.  Captures stack
effects as structured data, distinguishing effect-shaped block
comments (`( a -- b )`) from plain notes (`( pending refactor )`)
by looking for a free-standing `--`.

M2.3 — **resolve + emit**.  Three target kinds:

- `Builtin { vocab, factor_name }` — emit name bare, vocab in `USING:`.
- `QualifiedBuiltin { vocab, factor_name }` — emit `vocab:name` to
  dodge collisions.
- `UserDefined { factor_name }` — this compilation unit's `:` defs.

The collision dodging is important: `.` exists in both
`prettyprint` (Factor's debug printer) and `forth.runtime` (ANS
print-number).  `@` exists in both `math.ratios` (ratio
constructor) and `forth.runtime` (fetch).  Without the FQ emit,
Factor's parser errors with "resolves to more than one word".

Emit writes:

```
USING: forth.runtime io kernel math ;
IN: scratchpad
: square ( n -- n^2 ) dup * ;

5 square forth.runtime:. flush
```

The `IN: scratchpad` line was a late discovery — without it,
Factor errors with "Not in a vocabulary; IN: form required" the
moment we try to `:` define inside an eval'd string.  Scratchpad
is the conventional interactive playground vocab and exists in
any bootstrapped image.

### Test serialisation

cargo's test runner spawns each test in its own OS thread (for
panic isolation).  Factor's VM stores critical state in TLS that
gets initialised in `nf_init_factor`'s calling thread; eval from a
different thread crashes.

First attempt: keep everything single-threaded with
`--test-threads=1`.  Works but easy to forget.

Better: a process-wide `OnceLock<Mutex<()>>` gate that every test
acquires before touching the VM.  Tests still run on different
spawned threads (each creates its own VM), but only one VM is
alive at a time.  Tolerates a poisoned lock so a single panicking
test doesn't cascade — every test creates a fresh VM anyway, so
there's nothing shared to corrupt.

The user signed off on this:

> *"yep serialize all tests remember that, this is not thread safe,
> but forth never is, please continue"*

Forth is single-threaded by tradition.  Multithreaded Forth exists
(GForth has it) but it's outside our scope.

## Variable narrowing: the optimisation we'll write when M2.8 lands

Half-jibber-jabber while looking ahead to `VARIABLE`/`CONSTANT`.
The current `nf-addr` model is correct but coarse: every variable
allocates a byte-array wrapper and every `@`/`!` goes through the
FFI alien-accessor path.  Factor's compiler can't see across the
opaque memory load to optimise.

In practice ~95% of `VARIABLE x` declarations in ANS code use `x`
only as `x @`, `x !`, `x +!`.  The other 5% do address arithmetic
(`x cell+ @`), pass addresses across word boundaries, or otherwise
treat the address as a first-class value.

So after resolve, run a per-variable **escape analysis**:

```
For each VARIABLE x declared in the program:
  scan every reference to x in every body:
    if every use is immediately followed by @ or ! or +!
      → "narrow" — emit as a Factor SYMBOL, not an nf-addr
    else if x is dup'd, passed to a user word, has arithmetic
          done on it, or followed by an unknown sink
      → "escaping" — fall back to the nf-addr model
```

It's syntactic — no real flow analysis needed for the common case
because in Forth `x @` and `x !` are right-next-to-each-other
patterns at the token level.

The optimised emit looks like:

```
\ ANS                    Factor (narrow)
SYMBOL: mb-row
\ x !                    →  mb-row set-global
\ x @                    →  mb-row get-global
\ x +!                   →  mb-row [ + ] change-global
```

Factor's optimiser sees a real global cell.  It can fold
`get-global`, hoist across loops, value-propagate across `if`
branches.  The Mandelbrot demo's `mb-row` and `mb-rgb` win
because the inner pixel loop never touches the byte-array layer.

Same trick available for `CREATE`/`ALLOT` arrays in M2.9: narrow
to `specialized-array` when access pattern is indexed-only.
`FCONSTANT` is even better — it folds straight to a Factor
unboxed-float literal, which Factor's compiler propagates through
the IR.

The principle, across all of these optimisations, is the same:
**rewrite ANS idioms into shapes Factor's optimiser already
understands.**  Don't hide the meaning behind opaque pointer
manipulation if the user's intent is plainly a global variable, a
fixed array, a constant.

Same direction we'll take for the next big optimisation: when M2.5
lands DO/LOOP, the loop index `i` should be a Factor counted-loop
counter, not a return-stack push/pop, whenever the loop body
doesn't escape `i` to user words.  Counted-loop counters are
unboxed integers and live in registers; the return-stack-frame
path goes through fstack/slot-83 machinery.

I think this is going to be the recurring shape of every
performance win on this project: not "make Forth fast" but "make
Forth code look like the optimisable Factor it really is, then
let Slava's compiler do its thing."  Same compiler people pay
real money for in language X is sitting right there, sees a code
shape it knows, generates good code.

## What surprised me

**How small `forth.runtime` ended up.**  339 lines.  I'd budgeted
200–400 in the plan, hoped for the lower half, and made it.  The
memory model is 30 lines, return stack 25, do-loop 20, I/O 30 —
it's almost all glue.  The expensive line items I'd worried
about (a proper Forth assembler, peephole patterns, dictionary
machinery) just disappear when Factor owns the back end.

**How fast the front end came together once the IR target was
fixed.**  Lex+parse+resolve+emit is ~900 lines of Rust and four
hours of work.  Compare with WF32/WF64 where the parser was
maybe 2000 lines plus another 3000 of code generation.  Tells
you something about the right architectural choice eating most
of the apparent complexity.

**The `bi` order bug.**  I'd hand-traced `nf-!` three times
convinced it was right.  Only when the smoke test threw "expected
fixnum got byte-array" did I see the swap.  My internal model
of `[ p ] [ q ] bi` was right; my mental model of what
`set-alien-unsigned-cell` wanted on its stack was wrong.  Worth
remembering: when two systems' stack conventions disagree, you
WILL get it backward at least once.

**`IN: scratchpad` matters.**  I'd assumed eval'd source was
implicitly in some "interactive" namespace.  No — Factor's parser
is principled: any `:` definition needs a target vocab declared.
The error message ("Not in a vocabulary; IN: form required") is
clear enough, but it's the kind of thing that wastes a minute or
two of "what did I forget."
