## The success criterion

```
64 constant maxiter  maxiter 2 *  .   →   128
```

Pure constant folding through Factor's `CONSTANT:`.  Trivial, but
sets up the harder cases.

## Variable narrowing — what actually fires

Escape analysis is the load-bearing piece.  For each variable
declared via `VARIABLE x`:

1. Start with `escape[x] = Narrow`.
2. Walk every Expr list in the program (definition bodies,
   top-level runs, control-flow sub-blocks).
3. At each `WordRef("x")`, look at the next position in the
   same list:
   - `@`/`c@` → narrow sink (fetch).  No escape.
   - `!`/`c!` → narrow sink (store).  No escape.
   - `+!`     → narrow sink (increment).  No escape.
   - Anything else (literal next, control-flow keyword next,
     end of list, unknown word, user word, ...) → demote to
     Wide and record reason + span.

Conservative.  Variables not flagged narrow get the slow path,
which is correct (just slower).  Variables not referenced at
all stay narrow — vacuously every use is a narrow sink.

## Emit shapes

### Narrow

```
SYMBOL: x
0 x set-global   \ initialise to 0 (ANS variables read 0 pre-store)
```

References get the peep treatment:

| ANS          | Factor                        |
|--------------|-------------------------------|
| `x @`        | `x get-global`                |
| `x !`        | `x set-global`                |
| `x +!`       | `x [ + ] change-global`       |
| `x c@`       | `x get-global`                |
| `x c!`       | `x set-global`                |

(c@/c! mapped to the same Factor globals because Factor cells
hold values, not bytes — width discrimination for narrow vars
doesn't make sense.  Wide path keeps byte-level access.)

### Wide

```
SYMBOL: nf-var-x
<variable> nf-var-x set-global
: x ( -- addr ) nf-var-x get-global ; inline
```

References stay normal: `x` pushes the nf-addr (always the same
one), `x @` / `x !` flow through `forth.runtime:@` / `forth.runtime:nf-!`
as before.

## The signature gotcha on `+!`

Factor's `change-global` is `( variable quot -- )` — variable
BELOW, quotation ON TOP.  My first emit was `[ + ] x change-global`
which put quot below and symbol on top.  At runtime `change-global`
popped the wrong things: the symbol got passed where the quot
went, leading to "Generic word `call` does not define a method
for the word class.  Dispatching on object: n".

Fixed to `x [ + ] change-global`.  Signed the lesson: when
Factor's argument order disagrees with ANS's natural ordering,
get it backward at least once.  Same trick as the `bi` ordering
bug from day one.

## Pure-Rust sanity

The full Mandelbrot demo we have on disk uses `CONSTANT` (good —
M2.8 covers it) and `FCONSTANT` with computed values (`3.5e
240e f/ fconstant mb-dx`).  The computed form needs a small
compile-time evaluator — a Rust function that walks a `Vec<Expr>`
of arithmetic operations against int/float literals.  Cleanest
when only the basic ops (`+ - * / f+ f- f* f/ negate abs`)
appear, which is the realistic case.  Punted for now.

## Sema picks up new tables

`Sema::variables` and `Sema::constants` are populated during
`build`; the dump renders them with escape state visible:

```
Variables (1):
  n                def @ 1:22-1:32 [narrow]

Constants (1):
  maxiter          = 64         (Cell) @ 1:1-1:20

Escape analysis:
  n                narrow
```

When wide variables show up, the reason and source span get
shown next to the variable name.  Useful for debugging "why is
this getting the slow path?" without re-running.

## What this enables next

CREATE/DOES> in M2.9 will reuse the same escape-analysis
pattern, just with a richer set of recognised sinks (indexed
load/store rather than @/!).  Same insight: the user's intent
is "indexed array", emit it as Factor's specialized-array
where the compiler can see through.

FCONSTANT-with-computed-value also extends to general constant
folding inside other contexts: an `: pi 3.14159 ;` definition
where the body is a single literal is just a constant in
disguise.  We could fold those too.

Both are M2.9-or-later, but the path is clear.
