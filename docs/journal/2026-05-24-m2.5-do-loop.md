Three surprises today, each worth writing down.

## 1. `save-image-and-exit` zeros our special-object slots

Slot 82 (return stack) and 83 (loop-frame stack) are outside the
`save_special_p` range in `vm/objects.hpp`, which goes up to
`OBJ_BIGNUM_NEG_ONE = 81`.  When `build-image.sh` calls
`save-image-and-exit`, those slots are wiped to `f`.  Loading
the image and trying to access them gets you:

> Generic word `top>>` does not define a method for the
> POSTPONE: f class.  Dispatching on object: f

The init line `83 special-object [ 32 <fstack> 83
set-special-object ] unless` at parse time fires only during
build-image — by save-time it's done its job, by load-time the
slot has been zeroed.

Fix: lazy init inside the accessor.

```factor
: get-loop-frames ( -- fs )
    83 special-object
    [ 32 <fstack> dup 83 set-special-object ] unless* ; inline
```

`unless*` is "keep top if truthy, else run the quotation and
keep its result."  Cheap branch on a non-null slot; the
allocation path runs once per VM lifetime.

Same fix for slot 82 (return stack).

Could also have added the init to the startup quotation in
`build-image.sh`.  Lazy-in-the-accessor felt more local and
survives any future image-rebuilds without remembering to update
the script.

## 2. `inline` is non-negotiable for word-takes-quotation

First attempt at do-loop:

```factor
:: do-loop ( limit start quot -- )
    limit start push-loop-frame
    [ [ quot call bump-loop not ] loop ]
    [ ... cleanup ... ] recover ;
```

Loading this gets:

> The word do-loop cannot be executed because it failed to compile
> Cannot apply 'call' to an input parameter of a non-inline word

Factor's compiler refuses to compile `quot call` when `do-loop`
isn't `inline`.  The reason: the optimiser inlines the quotation
literal at every call site so the body's exact shape is visible
to effect inference and the JIT.  A non-inline `do-loop` would
have to dispatch on the quotation object at runtime, which the
compiler treats as an escape — and bans, for safety.

Same constraint propagates: callers that want effect inference
to see *through* `do-loop` (which is most callers — a `:`
definition without a stack-effect annotation can't be parsed
unless inference succeeds) also need `do-loop` inline.

The `inline` annotation is mandatory in this kind of word.  Cost:
each call site gets the body inlined into its compiled code.
Benefit: the JIT can specialise per call site and effect
inference works.  For a loop driver that's called a finite
number of times per program, the inlining cost is bounded.

## 3. LEAVE is harder than it looks

Three designs tried.

### v1: throw + recover

```factor
SYMBOL: loop-leave-tag
: leave ( -- ) loop-leave-tag get throw ;

:: do-loop ( limit start quot -- )
    limit start push-loop-frame
    [ [ quot call bump-loop not ] loop ]
    [ loop-leave-tag = [ ] [ rethrow ] if ] recover
    pop-loop-frame ; inline
```

Throws a sentinel tag.  Recovery checks the tag, runs nothing
on match.

**Bug:** Factor's `recover` unwinds the data stack to the saved
state at recover-entry.  That state is `( acc )` — the
accumulator value the body had BEFORE entering the protected
block.  All the accumulation across prior iterations is on the
data stack inside the loop, and recover restores it to its
pre-loop state.  Test expected 10; got 0.

### v2: with-return / return

```factor
USING: continuations ;
: leave ( -- ) return ;

:: do-loop ( limit start quot -- )
    limit start push-loop-frame
    [ [ quot call bump-loop not ] loop ] with-return
    pop-loop-frame ; inline
```

`with-return` captures a continuation; `return` invokes it.
Conceptually cleaner — no sentinel tag, no recovery check.

**Same bug.**  Factor's continuations save the data stack at
capture time and restore it on invocation.  Same problem as
recover: accumulator vanishes on LEAVE.

Both v1 and v2 are correct for control flow but wrong for ANS,
which treats the data stack as the program's PRIMARY state, not
something to be unwound around.

### v3: flag on the loop frame

```factor
TUPLE: loop-frame
    { limit integer }
    { index integer }
    { left? boolean } ;

: leave ( -- )
    get-loop-frames fstack-peek t >>left? drop ;

:: bump-loop ( delta -- done? )
    get-loop-frames fstack-peek :> frame
    frame left?>>
    [ t ]
    [ frame index>> delta + dup frame index<<
      frame limit>> >= ] if ;
```

LEAVE sets a flag on the innermost loop frame.  Bump-loop
checks the flag before bumping; if set, returns done?=true and
the loop exits cleanly.

**Deviation from strict ANS:** body code AFTER `LEAVE` in the
same iteration still runs.  ANS-strict LEAVE aborts the
iteration immediately.

In practice this is fine — the common ANS idiom is

```forth
do  body  some-cond if leave then  loop
```

with LEAVE at the end of an IF at the end of the loop body.
The deviation matters only when someone writes

```forth
do  some-cond if leave then  more-body  loop
```

and expects `more-body` not to run on the LEAVE iteration.  We
document it; if it ever bites a real user, we'll revisit.

### Why ANS-strict LEAVE is structurally hard

The data stack in ANS Forth *is* the program state.  Across
iterations of a loop, accumulators, running totals, partial
computations all live on the stack.  A non-local exit that
unwinds the data stack to a saved checkpoint destroys exactly
the state ANS code relies on persisting.

To do strict LEAVE properly, you'd need:

- a continuation that saves *only the call stack* and leaves
  the data stack alone, OR
- explicit save-stack-at-loop-entry / restore-stack-at-leave
  with a known delta accounting for what the body added.

Both are doable but each costs a non-trivial chunk of
infrastructure.  Pragmatic answer for now: ship the flag-based
LEAVE, document the deviation, defer the strict version to a
later milestone if a real user hits the edge case.

## A note on iteration cost

Today's flag-based LEAVE adds one slot read + one boolean test
per iteration.  In the inner-loop of a tight numeric kernel
(the Mandelbrot path coming in Phase 4) this is measurable.

The optimisation I have in mind for that: a separate `do-loop`
variant — let's call it `do-loop-fast` — for loops whose body
the compiler can prove doesn't contain `LEAVE`.  Same trick as
the variable-narrowing chat: when the static analysis can
guarantee LEAVE never fires, we don't need the flag check.  The
fast path is a plain counted loop.

That's a M2.7+ thing — after stack-effect inference lands and we
have the AST visibility to decide "does this body reach LEAVE."
Noted.

## Other small things

- Renamed AST `LoopKind` variants: `Plus1` (LOOP, step +1) and
  `PlusN` (+LOOP, body supplies step).  Read better than the
  earlier `Loop`/`PlusLoop`.
- `?do-loop` is `inline` for the same reason as `do-loop`.
- Resolve now knows I, J, LEAVE, UNLOOP.  UNLOOP maps to
  `pop-loop-frame` — discards the innermost loop frame.  Used
  by the upcoming `EXIT` (M2.10) to clean up before early
  return from a `:` definition.
- Parser's `do`/`?do`/`loop`/`+loop` enter the existing
  control-flow machinery; no new error variants needed, just
  added the words to the terminator lists.
