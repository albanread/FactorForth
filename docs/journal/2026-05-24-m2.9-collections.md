# 2026-05-24 — M2.9 standard collections: "ANS Forth for applications, not micros"

**Shipped**

- Standard defining-words: `array`, `farray`, `cbuffer`.  Each is
  a parser-recognised pattern: `<n> array <name>` defines a named
  collection with `( idx -- addr )` element-access semantics.
- forth.runtime gets `f@` / `nf-f!` for IEEE-754 doubles.
- `nf-addr+` rewritten with locals — the original `bi`-based form
  was applying its quotations to `n` (an integer) instead of
  `addr` (the nf-addr tuple), so the synthesised accessor crashed
  with "off>> does not define a method for fixnum."
- CREATE / ALLOT stays as the escape hatch but is no longer the
  recommended path.  Reframed in the docs.
- 4 new integration tests (phase29_*); total suite 87 lib + 30
  integration, all green.

**Open**

- 2D variants (`2array`, `2farray`).  Same pattern, two-index
  accessor.
- Record / struct shapes.  Probably wait until M2.9b's template
  machinery lands and we can express them via CREATE/DOES>.
- Optimisation: emit Factor `specialized-array`s for known-typed
  collections so Slava's compiler can vectorise.  Today everything
  goes through the byte-array path which is correct but opaque.

## The reframing the user pushed for

The original M2.9 plan in PLAN.md was "CREATE/DOES>: Trickier ANS
construct.  Emit a Factor parsing-word equivalent that allocates
data space at compile time and binds the runtime quotation."  That's
honest 1979.

The user's intervention was a clean three-step argument:

1. **CREATE/DOES> isn't really about byte-poking.** Strip away the
   dictionary memory and pointers and what remains is *higher-order
   function returns closure*.  CREATE = closure factory.  DOES> =
   closure body.  Each call instantiates a new function with private
   state.

2. **Most users never need that.** They need a typed array, a
   counted-string buffer, a 2D grid.  Those are common patterns
   that can be ready-made.

3. **So provide them as built-ins.** A user types
   `4 array primes` and gets an integer array.  No CREATE, no
   DOES>, no `cells allot`.

Then the slogan: **"we are ANS Forth for applications, not micros."**

That last line reframes the whole project's stance toward Forth's
microcomputer-era memory primitives.  We have them.  They work.
But they're not what we *want* the user to reach for.

## What's in the prelude now

Three defining-word patterns, recognised at parse time:

```forth
4 array primes              \ 4-cell integer array
240 farray xs               \ 240-cell IEEE-double array
80 cbuffer line             \ 80-byte character buffer
```

Each instance is `( idx -- addr )` — push an index, get the
address of that element.  Access via the corresponding store/fetch:

```forth
3 0 primes !  0 primes @     \ → 3                  (cells, @/!)
3.14 0 xs f!  0 xs f@        \ → 3.14               (doubles, f@/f!)
65  0 line c!  0 line c@     \ → 65 ('A')           (bytes, c@/c!)
```

The IR for `4 array primes` is:

```factor
SYMBOL: nf-coll-primes
32 <buffer> nf-coll-primes set-global
: primes ( idx -- addr )
    nf-coll-primes get-global swap 8 * forth.runtime:nf-addr+ ; inline
```

(`<buffer>` is forth.runtime's "make me an nf-addr around a zeroed
byte-array of N bytes."  `8 * nf-addr+` does the index → offset
arithmetic without exposing raw + on addresses.)

`farray` and `cbuffer` use the same template with the multiplier
flipped (8 for floats too, since we store doubles cell-wide; 1 for
bytes).

## The nf-addr+ bug

Surfaced the moment the first `array` test ran:

> Generic word `off>>` does not define a method for the fixnum
> class.  Dispatching on object: 0

The original definition:

```factor
: nf-addr+ ( addr n -- addr' )
    [ off>> + ] [ ba>> ] bi swap <nf-addr> ; inline
```

`bi` applies both quotations to the **top** of the stack — which
at entry is `n`, not `addr`.  Calling `off>>` on an integer is the
exact error message above.

The original `nf-addr+` had never been exercised before today —
narrow variables don't use it (their @/!/+! collapse to Factor
globals), and the wide-VARIABLE accessor doesn't either (it just
returns the static address).  Collections are the first user.

Fix is locals:

```factor
:: nf-addr+ ( addr n -- addr' )
    addr ba>> addr off>> n + <nf-addr> ; inline
```

Reads cleanly, no stack juggling.  The lesson is the same one that
keeps recurring: when stack-effect logic is anything more than
single-step, `:: locals` are clearer than `bi`/`tri` shuffles.
The journal has now mentioned this three times across as many
milestones; should probably be a paragraph in MANIFESTO.

## What this enables

The optimisation lever the day-one variable-narrowing chat
identified now extends naturally:

- **Variables** narrow to Factor `SYMBOL:` + `get-global`/`set-global`.
  Done in M2.8.
- **CONSTANTs** fold to literal `CONSTANT:` declarations.  Done in M2.8.
- **Collections** — for now they go through the wide nf-addr path
  with `swap N * nf-addr+`.  A future optimisation: detect "this
  array is only accessed with literal indices in a counted loop"
  and emit Factor's `specialized-array` instead.  Then the inner
  loop never touches our nf-addr layer and Slava's compiler can
  vectorise.

That last optimisation is what makes `farray` interesting beyond
ergonomics — Factor's `double-array` is a contiguous block the JIT
recognises.  Today's `nf-coll-xs` is opaque byte-array semantics.
Same data, but Factor can't reason about it.  Sub-milestone for
M2.9c or later.

## The Manifesto-aligned message

The shift in stance is worth stating clearly:

> NewFactor is ANS Forth for applications, not for microcomputers.
> The byte-level dictionary primitives (CREATE/ALLOT/HERE/`,`)
> remain available — every escape hatch the spec affords still
> works — but the *recommended* surface for new code is the
> typed collection vocabulary.  Users compose `array`, `farray`,
> `cbuffer`, and (eventually) `record` defining-words; they
> rarely if ever touch a raw address.  When they do, they're
> doing something genuinely novel and CREATE/DOES> is the right
> tool.  Most code isn't novel.

I should add that as a paragraph in MANIFESTO.md when I get a
chance.

## What surprised me

How short the implementation was.  ~150 lines of Rust to parse,
sema, and emit three collection types.  Tests included.  The
sema spine carried most of the weight — `Item::Collection` slots
into the same pipeline as Variable and Constant, no new
machinery.

The user's reframing happened in three short messages.  Each one
pointed at a higher level of abstraction than the one before:

- "CREATE/DOES> deeply interesting, separate it from the
  microcomputer mechanism"
- "model it at a much higher level"
- "we should create all the common use case... reduce the need
  for the memory poking in the first place"

Each step makes the design simpler.  Three messages = three
moves, each compositional.  Worth remembering: when the user is
challenging the design, they're usually pointing at an inversion
that makes the code smaller, not bigger.

## Pipeline state

```
ANS Forth source
  ↓  lex
  ↓  parse        ← recognises 4 array primes, 240 farray xs, 80 cbuffer line
  ↓  sema         ← collections table populated; effects (1 -- 1)
  ↓  emit         ← SYMBOL + buffer + accessor pattern
  ↓  eval
  → output
```

`compile()` produces a real IR that Factor accepts and executes.
26+4 = 30 integration tests including the new collection round-trips.
Wall-clock: ~2.7 s for the integration suite, ~0 s for lib.
