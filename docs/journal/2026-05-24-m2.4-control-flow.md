## The bug worth writing about: Factor's `while` hangs on `[ dup ]`

After implementing M2.4 the simple tests passed (`-5 abs .` → 5,
sign function, nested IFs).  Then I tried factorial via
`BEGIN/WHILE/REPEAT`:

```forth
: fact ( n -- n! )
    1 swap
    begin dup while
        tuck * swap 1 -
    repeat drop ;
5 fact .
```

The compiler emitted what looked obviously correct:

```factor
USING: forth.runtime io kernel math ;
IN: scratchpad
: fact ( n -- n! ) 1 swap [ dup ] [ tuck * swap 1 - ] kernel:while drop ;
5 fact forth.runtime:. flush
```

And it **hung Factor**.  Not a crash — just sat there, never
returning.  The first time I ran it I let cargo test run forever
before the user reminded me that "tests don't take 20 seconds —
that's failure."

Lesson 1: **every code path that touches the VM gets a watchdog.**
Test code, CLI tools, demos, everything.  No exceptions.

Lesson 2: when something hangs, bisect.  Don't sit there staring
at the IR.

### The bisection

Removed the `:` definition wrapper — same hang.  So it's the loop
itself, not the colon definition.

Tried the body once standalone (`5 1 tuck * swap 1 - . .`) —
worked fine, printed `0 5` (correct).  So the body is correct.

Tried Factor's documented minimal `while`:
`0 [ dup 3 < ] [ 1 + ] while .` — printed `3`.  So `while` works.

So the difference between working and hanging was the **predicate**.
My fact's pred is `[ dup ]` — extends the stack by 1 (adds a flag
copy) without consuming.  The working example's pred is
`[ dup 3 < ]` — consumes nothing (still dup'd) but the `<`
consumes both the dup'd copy AND its peer, leaving exactly one
flag.

So the *visible* effect of `[ dup 3 < ]` is `( a -- a flag )` —
adds a flag.  Mine is `( a -- a a )` — adds a copy that's
*used as* a flag.

Stack shape going into the body in both cases: pred-produced
state minus the flag.  Same shape.

But Factor's `while` definition has a sharp signature:

```factor
: while ( ..a pred: ( ..a -- ..b ? ) body: ( ..b -- ..a ) -- ..b )
```

The optimiser does abstract interpretation on `pred` and `body`.
With `[ dup ]` as pred, the inferred ..b is `..a x` (one larger
than ..a).  The body must then have effect `( ..a x -- ..a )` —
strip back to ..a.  My body changes shape from `( acc n )` to
`( acc' n' )` — same width but different element types as the
optimiser sees them.  The fixed-point search for the loop's
stack invariant never converges.  Compiler spins.

### The fix

Skip Factor's `while` entirely.  Use `loop` directly with an
explicit zero-test branch:

```
[ <pred> math:zero?
  [ f ] [ <body> t ] kernel:if
] kernel:loop
```

- pred runs, leaves flag on top of ..a
- zero? converts ANS-flag (0=false, non-zero=true) to Factor t/f
- if t (was 0): return f → loop exits
- if f (was non-zero): run body, return t → loop continues

`loop` only inspects the t/f returned by its quotation; it
doesn't try to infer effects across the whole pred/body pair.
The `if` branches both leave a single boolean on top, which is a
clean uniform shape.  Inference converges; eval finishes.

Trace for `5 fact`:

| iter | pre-pred | post-pred | post-zero? | branch | post-body  | quot returns |
|------|----------|-----------|------------|--------|------------|--------------|
| 1    | (1 5)    | (1 5 5)   | (1 5 f)    | else   | (5 4)      | t            |
| 2    | (5 4)    | (5 4 4)   | (5 4 f)    | else   | (20 3)     | t            |
| 3    | (20 3)   | (20 3 3)  | (20 3 f)   | else   | (60 2)     | t            |
| 4    | (60 2)   | (60 2 2)  | (60 2 f)   | else   | (120 1)    | t            |
| 5    | (120 1)  | (120 1 1) | (120 1 f)  | else   | (120 0)    | t            |
| 6    | (120 0)  | (120 0 0) | (120 0 t)  | then   | (120 0)    | f            |

Loop exits with stack `(120 0)`.  `drop` removes the 0.  `.` prints
120.  ✓

### Why this is a recurring shape

I suspect Factor's optimiser-friendly forms will keep showing up
as a constraint on our emit.  The user-written ANS Forth has
idioms that are correct under ANS semantics but produce stack
shapes that the Factor optimiser doesn't recognise as the
patterns it can handle.  When we emit IR, we have to translate
into Factor's *idioms*, not just Factor's *syntax*.

The variable-narrowing optimisation we sketched yesterday is the
same principle: emit `SYMBOL: x` + `get-global` not because that's
faster than `nf-addr` + `@`, but because **Factor's compiler can
SEE INTO the SYMBOL form** and optimise across it.  Same here:
emit `[ pred zero? if-clause ] loop` not because it's faster than
`while`, but because the effect inference completes.

I'll keep watching for this pattern.  When something Factor
ought to handle silently turns into a compiler hang, the answer
is almost always "express it differently."

### Watchdog as default

The user's exact words:

> *"the actual test runner also needs to bloody time out and fail
> tests that hang... tests that take 20 seconds have failed."*
>
> *"dont be a pratt and write tests that hang forever LOL"*
>
> *"Always add a watchdog"*

Taken to heart.  Watchdog is now in:

1. `tests/smoke_runtime.rs::with_vm` — auto-armed for every
   integration test, 20 s ceiling.  Label comes from
   `std::thread::current().name()` so the cargo output points at
   the offending test by name.
2. `src/bin/embed_smoke.rs::main` — auto-armed at program start,
   15 s default, `EMBED_SMOKE_TIMEOUT=N` to override.  Means
   any shell invocation that hangs Factor returns control in 15 s
   max.

`std::process::abort()` is the mechanism — not `panic!`, not
`exit(1)`.  Abort terminates the process *immediately* without
running drop handlers, which is what we want when Factor is
stuck inside an FFI call we can't unwind.  The exit code is
non-zero so cargo test sees failure.

When I write Phase 3's session module, the watchdog moves there
and becomes a per-eval timeout instead of per-process abort,
which lets the host recover from a hung evaluation by tearing
down just the worker thread.  But that's later.

## Other small things

* Added comparison and bitwise builtins to the resolve table
  (`<`, `>`, `=`, `<>`, `<=`, `>=`, `0=`, `0<`, `0>`, `and`,
  `or`, `xor`, `invert`).  ANS `AND/OR/XOR/NOT` are bitwise; we
  emit `bitand`/`bitor`/`bitxor`/`bitnot`.  ANS `NOT` was
  removed in the 2012 standard anyway; only `INVERT` is current.
* The `IN: scratchpad` line on every emit makes `:` definitions
  legal.  Without it Factor errors "Not in a vocabulary".  Found
  by reading the error message and remembering to.
* `.` and `@` collide with prettyprint / math.ratios; we emit
  fully qualified.  The `phase24_nested_if_in_else` test was a
  good exercise of that — `." neg" ." zero" ." pos"` strings
  through the `."` emit (`"text" forth.runtime:type`).
