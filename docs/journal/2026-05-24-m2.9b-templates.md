# 2026-05-24 — M2.9b CREATE/DOES> as templates

**Shipped**

- `: myarray create cells allot does> swap cells + ;` parses as
  an `Item::Template`, not an `Item::Definition`.  At sema time,
  `<n> myarray <newname>` triples in TopLevel are folded into
  `Item::TemplateInstance` items.  Emit produces the SYMBOL +
  buffer + accessor pattern with the user's `does>` body
  inlined.
- `+` inside a `does>` body is translated to `nf-addr+` at emit
  time so the user's `swap cells +` lands on our nf-addr.
- `allot`/`here`/`floats` added as runtime stubs (allot drops,
  here returns 0) — these are mostly parser markers but
  programs that reach for them at runtime get a quiet no-op
  rather than an error.
- Two integration tests; 90 lib + 40 integration total, all
  green.

**Open**

- Constructors more complex than `cells allot` / `chars allot`.
  Specifically `,` (comma) for initial-value cell stores —
  needed for lookup-table templates.
- DOES> bodies with `-` (address subtraction) — currently only
  `+` is translated; subtraction would need `nf-addr-`.
- Template effect inference: instances get `(1 -- 1)` as a
  default approximation; should be derived from the does_body.
- Mutual-recursion DEFER: — would let templates call each other
  or call later-defined words.

## Implementation walk-through

The pipeline change:

```
parse:
   "...: foo CREATE ... DOES> ... ;..."  →  Item::Template
   "...10 foo bar..."                    →  TopLevel { 10, WordRef("foo"), WordRef("bar") }

sema::expand_templates_pre_resolve:
   walks every TopLevel; folds `<n> template-name <newname>`
   triples into Item::TemplateInstance placed BEFORE the
   surrounding TopLevel (so newname is declared in source order).

resolve:
   sees a normal program with newname registered as a user word.

emit:
   Item::Template → emit nothing (it's compile-time only)
   Item::TemplateInstance → SYMBOL + buffer + accessor, with
                            the captured does_body, with `+`
                            translated to nf-addr+.
```

## The two ordering bugs (caught fast)

### Bug 1: `expand_templates` ran AFTER resolve

First version: I'd put expand_templates as a post-resolve pass
in sema::build.  But resolve walks every TopLevel WordRef and
errors on unknown names.  The `ages` reference in the user's
`42 2 ages !` was unknown at that point — because expand hadn't
run yet to introduce `ages` as a user-dictionary entry.

Symptom: `newfactor: unknown word \`ages\` at line 1, column 62`.

Fix: pulled expand into a standalone pre-resolve pass:

```rust
pub fn build(program: Program) -> Result<Sema, ResolveError> {
    let program = expand_templates_pre_resolve(program);
    let resolved = resolve(program)?;
    ...
}
```

Order matters.  Pipeline now: parse → expand-templates →
resolve → effect → escape → analyse-call-graph.

### Bug 2: row-vars annotation vs concrete inferred effect

First emit used `( ..a -- ..b )` for the instance's stack effect
annotation.  Factor's compiler inferred `( idx -- addr )` from
the body and rejected the mismatch:

> The word ages cannot be executed because it failed to compile
> Stack effect declaration is wrong
> inferred ( x -- x )
> declared ( ..a -- ..b )

Factor accepts row-var declarations on definitions WITHOUT
bodies (it's the most-general signature), but when there's a
concrete body whose effect can be inferred, the annotation has
to match.  Fixed to emit `( idx -- addr )`.

This is also the same lesson from M2.10 (effects-as-warnings):
synth from the body is more believable than what we declare.
Here we should probably *derive* the annotation from the
does_body's effect rather than hardcoding `( idx -- addr )`,
since templates with weirder does bodies will trip the same
issue.  Sub-milestone.

## What the user can write

```forth
\ Define your own indexed-array shape.
: myarray  create cells allot  does> swap cells + ;

\ Use it.  No CREATE, no ALLOT in the user's day-to-day code.
4 myarray ages
4 myarray prices

100 0 ages !       \ ages[0] = 100
200 0 prices !     \ prices[0] = 200
0 ages @ .         \ → 100
0 prices @ .       \ → 200
```

The two instances have INDEPENDENT state — each gets its own
SYMBOL + buffer.  Aliasing across instances was a real fear
before I checked; it's correctly not a problem because each
`Item::TemplateInstance` emits its own `nf-tmpl-<name>` SYMBOL.

## Architecture observation

This is the most invasive Phase 2 change since the sema
refactor — it touched parser (new return type for
colon_definition), AST (two new variants), sema (new pre-pass,
two new BTreeMaps, walks in three places), emit (two new
emit_* functions plus the +/nf-addr+ translator).

Yet it still came in around 200 lines of Rust net plus a few
forth.runtime additions.  The sema spine carried most of the
weight — every new walker just got an additional match arm.

What surprised me: how naturally the "compile-time-of-the-caller
becomes a program-level transformation" mapped onto the AST.
ANS treats template instantiation as a runtime side-effect of
calling the defining-word; we treat it as a static expansion
that produces a new program item.  Same observable behaviour,
much cleaner static analysis.

The journal had been observing for a few entries that each new
ANS surface looks weird until you reframe it as a DSL or a
template or a higher-order function.  This one is the
canonical case: CREATE/DOES> *is* template instantiation, and
once that frame is in place the code writes itself.

## Pipeline state

```
ANS Forth source
  ↓ lex             ✓
  ↓ parse           ✓ Item::Template detected on `:` close
  ↓ expand-templates ✓ pre-resolve: `<n> name <newname>` → Item::TemplateInstance
  ↓ resolve         ✓ instance names registered as user dictionary
  ↓ effect          ✓ instances get (1 -- 1)
  ↓ escape          ✓
  ↓ analyse         ✓ call graph / use sites
  ↓ emit            ✓ Template → nothing; Instance → SYMBOL + buffer + accessor
  ↓ eval            ✓
  → output
```
