# M3.0.3 #41 — Rust-driven Forth 2012 test runner (2026-05-24)

The proving ground.  Canonical ANS T{ <code> -> <expected> }T
assertions actually running against NewFactor's pipeline.

## The pivot

Original plan: load the canonical Hayes / Gerry Jackson tester.fr
as Forth source and let it drive itself.  Reality check uncovered
hard blockers:

- `tester.fr` uses `SOURCE`, `>IN`, `WORD` — Forth's interactive
  parser-state introspection, which NewFactor's compile-then-eval
  pipeline doesn't expose
- `?DUP` (effects-polymorphic) — Factor's static inference can't
  type it, filed in #45
- `[CHAR]`, `IMMEDIATE`, `\` parser-word — not shipped
- The whole architectural assumption (interactive Forth top-level
  with mutable source pointer) is fundamentally different from
  our model

User's idea: **treat the test files as DATA.  Have Rust run them.**

This sidesteps everything.  Rust parses the `T{ -> }T` shape
itself, drives each assertion as two independent evals (code +
expected), and compares the resulting stack dumps as text.

## Implementation

`tests/session_test_runner.rs` is a self-contained ~340-line
runner with:

1. **Comment stripper** (`strip_comments`): preserves newlines for
   line-number tracking; handles `\` line comments and `( ... )`
   block comments per ANS space-delimited rules.

2. **Assertion extractor** (`extract_assertions`): tokenises on
   whitespace, finds `T{` / `->` / `}T` markers (case-insensitive),
   slices out (code, expected) pairs.  Multi-line assertions
   work because the tokeniser walks linearly across newlines.

3. **Per-assertion runner** (`run_assertion`): for each test,
   runs both `<code>` and `<expected>` followed by the same
   stack-dump tail `begin depth 0> while . repeat` against a
   fresh-eval on the shared Session.  Compares the trimmed
   captured outputs as strings.

4. **Pass / Fail / NYIMP tally**: NYIMP fires when compile()
   returns an unknown-word error — distinguishes "we haven't
   built this word yet" from "the answer is wrong".  Watchdog
   in Session::eval (20s) catches infinite loops.

## What's deferred

The corpus is a hand-curated subset.  The full canonical test
files (`E:/wf32/forth2012-test-suite-master/src/core.fr`, etc.)
have assertions for words we haven't shipped:
- `?DUP` (#45 — Factor inference incompatible)
- `[CHAR]` `CHAR` (parsing words)
- `U<` `U>` `COUNT` `KEY?` `EXIT` `.S` `MOVE` `PICK` `ROLL` (#46)
- `DEFER` `IS` (#49)
- File I/O beyond INCLUDED
- Source-input introspection (`SOURCE`, `>IN`, `WORD`, `EVALUATE`)

Each missing word would show up as NYIMP when the corpus expands
to include its tests.  The runner already classifies them
correctly.

The CANONICAL `core.fr` also depends on `tester.fr`'s state-
machine words.  Adopting `core.fr` verbatim requires either
implementing the source-input model OR rewriting `core.fr`'s
assertions in a form our parser-state-free runner accepts.
The latter is the pragmatic path: grow `ans-core-corpus.fs` by
hand, cherry-picking from `core.fr` line-by-line.
