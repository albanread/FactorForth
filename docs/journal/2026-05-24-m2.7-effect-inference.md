## Design

`Effect` is just `(inputs, outputs)` as `u32`s, plus an `Unknown`
variant for cases the analyser can't or won't reason about.

```rust
pub enum Effect {
    Known { inputs: u32, outputs: u32 },
    Unknown,
}
```

Composition is `a.then(b)`:

```
( ai -- ao ) then ( bi -- bo )
  if ao >= bi:  ( ai -- ao - bi + bo )           // b consumes from a's output
  else:         ( ai + (bi - ao) -- bo )         // b dips into a's pre-state
  Unknown if either side is Unknown
```

Walking a body composes effects left-to-right.  At the end, the
result is the body's net effect.

For each kind of expression:

| Expr             | Effect                                         |
|------------------|------------------------------------------------|
| Int / Float lit  | (0 → 1)                                        |
| `."` string      | (0 → 0) (emits at runtime via `type`)           |
| `S"` string      | (0 → 2) (c-addr u on top)                       |
| `C"` string      | (0 → 1) (counted-string c-addr)                 |
| WordRef builtin  | Looked up in `builtin_effects()` table          |
| WordRef user     | Looked up in `user_effects` map                 |
| Control flow     | Unknown (for now)                               |

For declarations:
1. Pass 1: seed `user_effects` with declared effects (or Unknown
   for undeclared definitions) so mutual recursion has a base.
2. Pass 2: walk each body, compose effects, compare against the
   declared if any.  Errors go into a list.

The first error is returned by `compile()`; subsequent ones often
cascade and would just clutter output.

## What's coming for control flow

Each control structure has a known formula that's an exercise for
the next sub-milestone.  Sketched here so we don't lose them:

### IF / THEN (no else)

The flag is consumed; the body must have effect `( n -- n )` (no
net change) for the join point to make sense — both paths (body
taken, body not taken) must converge to the same shape.

```
IF/THEN  effect:  ( 1 + body_in -- body_out )     where body_in == body_out
```

Body inference + a check that `inputs == outputs`.  If not, that's
an effect error at the IF.

### IF / ELSE / THEN

Both branches must have the same effect.  Total: consume the flag,
then either branch.

```
IF/ELSE/THEN  effect:  ( 1 + br_in -- br_out )    where then.effect == else.effect
```

### BEGIN / UNTIL

Body produces a flag (`body : a -- a 1`).  Loop iterates while
flag = 0.  Total: `( body_in - 1 -- body_out - 1 )` per the loop
invariant `body_out == body_in + 1`.

For ANS net: `( 0 -- 0 )` typical (the body restores its consumed
state on each iteration plus a flag).

### BEGIN / WHILE / REPEAT

Pred: `( a -- a 1 )` (produces flag, preserves base).  Body: `( a
-- a )`.  Loop: `( pred_in - 1 -- pred_in - 1 )` after the flag
consumption.

### DO / LOOP

Consume limit + start (2 in).  Body is `( a -- a )`.  Net `( 2
-- 0 )` plus the body's contributions over iterations (which must
balance).

### CASE / OF / ENDOF / ENDCASE

Dispatch value consumed (1 in).  Each arm has the same `( a -- b
)` effect — including the default branch.  Total `( 1 + a -- b
)`.  If arm effects don't unify, that's an error.

All of these can be expressed as small functions in `effect.rs`
following the same `infer_block` pattern.

## Variable narrowing got closer

Remember the variable-narrowing chat from day one?  The escape
analysis it depends on becomes a one-pass walk *after* effect
inference: for each `VARIABLE x`, check whether every reference
to `x` is immediately followed by `@`, `!`, or `+!`.  Effect
inference doesn't directly enable that — but it gives us the
*infrastructure* (a per-expression walker, builtin table,
per-definition results) that the narrowing analysis can plug
into.  M2.8 will inherit it for free.

## What surprised me

How much shorter the implementation was than I expected.  ~300
lines of Rust for the analyser + table + tests, including
docstrings.  The compositional algebra of effects (`a.then(b)`)
is one of those cases where the math is so obviously the right
shape that the code is just transcription.

The borrow checker poked me twice — once where Env held
`&user_effects` and I wanted to update it after `infer_block`
returned.  Solution: rebuild Env per iteration so the borrow
ends before the mutate.  Standard Rust shape, one minute to fix.

The tests almost wrote themselves.  Composition tests, simple
correct programs, the canonical mismatch, propagation through
user words.  Nine unit tests + a docstring's worth of comments
and we have decent coverage.

## What I deferred and why

- **Control-flow effects**: each structure has a formula but
  writing them right means handling effect-join (both branches
  of an IF must unify, all arms of a CASE must unify, etc.).
  That's a clean follow-up; punting now to ship M2.7's
  user-visible benefit.
- **Floating-point stack**: ANS Forth has a separate FP stack
  (`F:` items in effect annotations).  Not modelled yet.  Most
  programs only use the data stack.
- **Return stack**: `>R` and `R>` move items between stacks.
  Untracked.  Could be added with another counter pair.
- **Recursion**: a self-calling word with no declared effect
  currently gets `Unknown`.  Fixpoint inference is doable but
  not yet needed.
- **Type-or-purpose item names**: declared effect comments can
  carry names (`( n -- n^2 )`).  We ignore them for inference;
  they're metadata only.
