# Phase 3.1b — host callbacks captured end-to-end (2026-05-24)

All five `tests/session_io.rs` tests pass.  The FFI chain

```
ANS user code → Factor word (emit / cr / type / `.`)
              → Factor's `write` → output-stream get → stream-write
              → nf-host-output-stream method → rt_write_char  (Rust extern)
              → CURRENT IoState → captured Vec<u8> / stdout / GUI callback
```

is fully connected.  KEY closes the loop in the other direction
(rt_read_char ← input_q).

## Two non-obvious problems we hit

### 1. Symbol lookup: factor.dll vs the host .exe

Factor's `add-library "nf-host" "<path>" cdecl add-library` followed
by `FUNCTION:` ultimately calls `ffi_dlsym(dll, "rt_write_char")`.
The patched embedding API had `hFactorDll` pointing at *factor.dll
itself* (so primitive lookups during `init_factor` succeed) — fine,
except that means `dll == NULL` fallbacks went to factor.dll, not
the host .exe whose export table actually has `rt_write_char`.

Fix in `vm-build/vm/os-windows.cpp`: store an additional
`hHostExe = GetModuleHandle(NULL)` during `init_ffi`, and in
`ffi_dlsym` fall back to it when the factor.dll lookup misses:

```cpp
cell factor_vm::ffi_dlsym(dll* dll, symbol_char* symbol) {
  if (dll) return (cell)GetProcAddress((HMODULE)dll->handle, symbol);
  cell r = (cell)GetProcAddress(hFactorDll, symbol);
  if (r == 0 && hHostExe && hHostExe != hFactorDll)
    r = (cell)GetProcAddress(hHostExe, symbol);
  return r;
}
```

The user spotted this faster than I did — *"we compile the factor
dll right?"* — pointing at the patched VM rather than a workaround.

### 2. `with-string-writer` shadows our global output-stream

After install-host-streams worked and a direct test
(`"--ABC--" output-stream get-global stream-write`) produced
rt_write_char calls, *user* code (`65 emit 66 emit 67 emit cr`)
still didn't.

Cause: Factor's stock `eval-callback` (basis/eval/eval.factor) is

```factor
[ eval>string utf8 malloc-string ] alien-callback
```

and `eval>string` does its work `with-string-writer`, which uses
`with-variable` to **dynamically rebind** `output-stream` for the
duration of the eval.  Our `install-host-streams` only set the
*global* binding — the dynamic shadow won.

Fix: add a sibling word that uses `set` (not `set-global`):

```factor
: rebind-host-streams ( -- )
    <nf-host-input-stream>  input-stream  set
    <nf-host-output-stream> output-stream set ;
```

and prepend `USING: forth.runtime ; rebind-host-streams\n` to every
eval source from the worker thread.  That overrides the
with-string-writer frame from inside, so user output flows through
rt_write_char.

The captured-string return from `nf_eval_string` still works — it
captures whatever's left of the with-string-writer scope, which is
now empty unless the user explicitly writes to it (errors do, via
`print-error`).

## Lessons for future I/O hooking

* When Factor's eval-callback is the entry point, **every dynamic
  rebinding lives inside `with-string-writer`'s shadow**.  Reach
  for `set` over `set-global` when you need to override a stream.
* The two-handle FFI gotcha (`hFactorDll` ≠ host module) only bites
  embedders.  When the host has its own exports to share, the
  fallback is mandatory.
* Symbol-keeper statics: Rust's `extern "C" fn` won't survive
  link-time dead-code elimination just because of `#[no_mangle]`
  — add `#[used] static _KEEP: extern fn(..) = rt_*;` per symbol.
* `/EXPORT:` linker args via build.rs must target *only* the bins
  that actually contain the symbol, or the embed-smoke link fails.
