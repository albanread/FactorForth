# M2.x #44 — LET DSL ported from WF64 (2026-05-24)

The headline feature.  Infix algebra inside postfix Forth, with
Factor's compiler doing the unboxing.  Shipped end-to-end in one
pass — straight-line LET, WHERE clauses, function calls all
work together.

## Surface

```text
LET ( in1, in2, ... ) -> ( out1, out2, ... ) =
    result1, result2, ...
    WHERE name1 = expr1
    WHERE name2 = expr2
    ...
END
```

Supports:
- Arithmetic `+ - * /`, unary minus, `**` (right-associative)
- Comparisons `< > <= >= == !=` returning `1.0` / `0.0`
- Parens for precedence override
- Function calls: `sqrt abs min max floor ceil round trunc`
  (SSE-direct) and `sin cos tan asin acos atan atan2 exp log pow
  hypot` (libm via Factor's `math.functions`)
- WHERE clauses (executed in source order; topological sort
  deferred to #50)
- Constants `pi`, `e`
- `\` line comments and `( ... )` block comments inside the block

## The lowering

The Mandelbrot iteration kernel

```forth
LET (z_re, z_im, x, y) -> (z_next_re, z_next_im, mag) =
    re, im, rmag
    WHERE re   = z_re * z_re - z_im * z_im + x
    WHERE im   = 2 * z_re * z_im + y
    WHERE rmag = re * re + im * im
END
```

lowers to (with our `nfl-` prefix mangle):

```factor
[| nfl-z_re nfl-z_im nfl-x nfl-y |
    nfl-z_re nfl-z_re math:* nfl-z_im nfl-z_im math:* math:- nfl-x math:+ :> nfl-re
    2.0 nfl-z_re math:* nfl-z_im math:* nfl-y math:+ :> nfl-im
    nfl-re nfl-re math:* nfl-im nfl-im math:* math:+ :> nfl-rmag
    nfl-re nfl-im nfl-rmag
] call( nfl-z_re nfl-z_im nfl-x nfl-y -- nfl-z_next_re nfl-z_next_im nfl-mag )
```

Factor's compiler:
1. Recognises the `[| | ]` quotation as a lexical-locals form.
2. Inlines the `call( ... )` (it's a known quotation).
3. Type-infers the floats from the input bindings.
4. Unboxes the floats through the entire arithmetic chain.
5. Emits straight-line XMM-register asm for the body — same
   shape WF64's hand-written MASM would produce.

**The substrate bet pays off.**  WF64 spent ~900 lines on LLVM-MC
codegen (XMM register allocation, libm trampolines, A-normal-form
conversion, sign-bit XOR for `Neg`).  We threw all of that away
and got equivalent codegen for free because Factor's optimiser
was designed to do exactly this kind of work.

## Architecture

```
src/compiler/let_lang/
  mod.rs       —  re-exports parse() and lower_to_factor()
  parser.rs    —  ported verbatim from WF64 (~500 lines)
  codegen.rs   —  new (~200 lines)
```

The main lexer captures `LET ... END` as a single `Tok::LetBlock`
containing the raw text.  The parser sees that token and calls
`let_lang::parse()` to produce a `LetForm`.  The AST node is
`Expr::LetForm { form, span }`, treated like any other expression
by resolve / effect / sema (it's self-contained — no external
word refs, fixed input/output arity for effect inference).

At emit time, `Expr::LetForm` is lowered via
`let_lang::codegen::lower_to_factor()`.

## The two-bug debug

### Bug 1: `LetForm` derive PartialEq

`Expr` derives `PartialEq` but my initial `LetForm` only derived
`Debug, Clone`.  Adding `PartialEq` to `LetForm` and its `Expr`
sub-type fixed it.

### Bug 2: `math.functions` not in USING

The lowered IR uses fully-qualified `math.functions:sqrt` etc.
Factor's parser STILL requires the vocab to appear in `USING:`
even for `vocab:name` qualified references — the qualified form
is a parser shortcut, not a way to bypass vocab loading.

Fix: add `math.functions`, `math.order`, `math.constants` to the
baseline `vocabs_needed` set in `emit.rs`.  Cheap (one entry
each, present in any basis-bootstrap Factor image) and unlocks
LET's full function-call set.

## What's deferred

- **#50 topological sort of WHERE clauses**: user code currently
  must write WHEREs in dependency order.  Cycles and forward refs
  produce a runtime "undefined name" error from Factor's parser
  rather than a compile-time diagnostic.  WF64's spec calls for
  topo-sort + cycle detection; mine does neither.  All current
  use-cases (including Mandelbrot) write WHEREs in order, so this
  is a polish item, not a blocker.
- **Shared float-buffer integration** (#37): the natural next
  step is to have LET that writes outputs DIRECTLY into a host-
  allocated `f64[]` (vertex buffer, audio sample stream) rather
  than back to Factor's data stack.  That sidesteps the heap-
  float boxing on the output side entirely.

## What this unlocks

**The whole graphics path.**  The Mandelbrot benchmark milestone
(#10) can now actually be written — the inner-loop kernel is a
LET form that Factor will compile to tight XMM code.  Comparing
that against WF64's hand-rolled MASM is the v1 proof point.

**Audio / DSP.**  Filter coefficients, wavetable lookups, FIR
kernels — all infix-algebra-heavy code.

**Physics / simulation.**  Vector math written naturally.

LET wasn't on the original ANS roadmap; it's the kind of thing
that becomes possible when the substrate is doing work for you
instead of fighting you.  WF64 ships ANS conformance plus this
DSL; NewFactor ships ANS conformance plus this DSL with 1/5 the
implementation effort because Factor's compiler is the DSL's
backend.
