# 2026-05-28 — SEE: compile-time word introspection

Rounding out the Programming-Tools set after `.S` / `WORDS` /
`DUMP`.  The user's framing set the scope: "We need to consider
what SEE can usefully show ... lots of useful attributes of a
'word' it can display, but probably not the actual definition,
although I guess it could if we kept the last compiled source
code as well.  Which may not be a big deal really."

It wasn't a big deal.  We keep the source, and `SEE` shows it.

## What SEE shows

```
SEE square
  square   colon definition   ( n -- n2 )
  source:
    : square ( n -- n2 ) dup * ;

SEE dup
  dup   builtin   ( a -- b c )
  factor: dup

SEE answer
  answer   constant   value: 42
  source:
    42 CONSTANT answer

SEE point
  point   class
  slots: x y
  source:
    CLASS: point SLOT: x SLOT: y ;
```

For a user definition: the kind, the declared stack effect, any
extra detail (a constant's value, a class's slots + parent), and
the verbatim original ANS source.  For a builtin: the "builtin"
tag, its arity, and the Factor word it maps to.  For an unknown
name: a graceful "not defined this session" note rather than a
compile failure.

## The key architectural decision: SEE is a compile-time word

`.S` / `WORDS` / `DUMP` are runtime words that introspect the
live Factor stack or image.  `SEE` is different — everything it
wants to show (the ANS source, the declared effect with parameter
names, whether a name is a generic vs a constant) lives in our
**Rust front end**, not in the Factor image.  The Factor word
object only carries a name, an ANSI-mangled effect, and a vocab;
decompiling it needs prettyprint, which the slim image lacks.

So `SEE` is a **parsing word**, exactly like `'` (tick) and `TO`:
it consumes the next token at parse time, and at *emit* time the
compiler looks up everything it knows about that name and lowers
the whole report to a single literal print
(`"...text..." print-string`).  SEE therefore costs nothing at
runtime beyond the print itself — all the work happens in the
compiler.

This is the same "we're a compiler, not a translator" identity
that's shaped the whole project: the rich knowledge is on our
side, so the introspection word reaches into our side.

## Keeping the source

A new `CompileContext.docs` map holds one `WordDoc` per defined
name — kind, rendered effect, the source substring (sliced from
the original text via the span we already track), and an optional
detail line.  It persists across evals like the other metadata
(`classes`, `values`, `templates`), so `SEE foo` works on a `foo`
defined in an earlier REPL line.  Newest definition wins, matching
Factor's redefinition semantics.

The cost is what the user predicted: a few KB of strings per
session.  Negligible.  And showing the source the user actually
wrote is strictly more faithful than reconstructing it from the
emitted IR — no decompiler to get subtly wrong.

## What changed

- **`src/compiler/ast.rs`** — `Expr::See { name, span }` (a
  parsing-word expr, like `Tick` / `To`); `WordDoc` struct.
- **`src/compiler/parse.rs`** — recognise `see` as a parsing
  word that consumes the next token.
- **`src/compiler/{resolve,sema,effect,dump}.rs`** — `Expr::See`
  arms (resolve: no word-ref; sema: records a use-site so a
  SEE-only word isn't flagged dead; effect: `( -- )`; dump:
  debug print).
- **`src/compiler/mod.rs`** — `CompileContext.docs`; `build_docs`
  walks the AST + slices source into `WordDoc`s; the driver
  seeds `sema.docs` (prior + this compile) and persists back.
- **`src/compiler/emit.rs`** — `emit_see` builds the report
  (user word from `docs`, builtin from the now-`pub`
  `resolve::builtin_table` + `effect::builtin_effects`) and
  lowers it to a literal print.
- **`tests/diag_see.rs`** — 6 tests: colon-def-with-source,
  builtin-factor-target, constant-value, class-slots,
  cross-eval, unknown-graceful.

## Scope held to "attributes + source"

Methods are intentionally skipped in the doc store: a `METHOD:`
shares its generic's name, so recording it would clobber the
generic's entry.  `SEE genericname` shows the generic; listing a
generic's methods + specialisers needs a generic→methods map,
which is a clean follow-up but more plumbing than this cut
warranted.
