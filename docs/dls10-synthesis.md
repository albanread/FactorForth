# DLS '10 paper — synthesis notes

Working notes from reading *"Factor: a dynamic stack-based programming language"*
(Pestov / Ehrenberg / Groff, DLS 2010, 12 pages, downloaded to
`docs/upstream/external/papers/dls10-pestov-ehrenberg-groff.pdf`).

Audience: next-session-Claude (and anyone resuming this project cold).
Section refs are to the paper unless otherwise noted.

---

## 1. The five claims that matter most

These are the load-bearing facts for our mission, restated in our terms:

1. **The VM contains a "base compiler" written in C++.  The "optimising
   compiler" is written in Factor and lives in the image.**  (§3.1, §3.3, §3.4)
   So there are TWO compilers, not one, and they sit in different parts of
   the system.  This changes everything about the slim-image task — see §5
   below.

2. **Every quotation, including REPL input, is compiled before being called.
   There is no interpreted execution mode.**  (§3.1: "All source code is
   compiled into machine code by one of two compilers"; §5: "the parser
   always compiles code into quotations.  Even code entered at the top-level
   is first compiled into a quotation, and then the quotation is immediately
   called and discarded.")  This means our Rust front-end's output IS the
   compiler's input — we don't go through any text reader.

3. **The stack checker is part of the type system, not a lint pass.**  (§2.1.3,
   §2.1.4)  Words with undeclared stack effects are rejected at compile time
   unless wrapped in `call( -- )` or `with-datastack`.  Row-polymorphic
   combinators must inline their quotation arguments — failing to inline is
   a compile error, not a missed optimisation.  This is a real design
   constraint for any ANS Forth front-end (see §6 below).

4. **Compiled code is movable.  GC compacts the code heap, including
   relocating jump targets in the callstack.**  (§3.1: "Even compiled code
   is subject to compaction… To support early binding, the garbage collector
   must modify compiled code and the callstack to point to newly relocated
   code.")  Implication: we cannot hold long-lived raw pointers to JITted
   bytes from Rust.  We address compiled code via Factor's word/quotation
   handles only.

5. **Word redefinition uses early binding plus dependency tracking, not late
   binding plus dictionary lookup.**  (§3.2)  The compiler records what
   classes/layouts/methods a word's compiled body assumed; on redefinition,
   the inverted index identifies which callers need recompilation.  An
   "incremental linker" rewrites direct call sites to the new address.
   Forth's interactive redefinition becomes essentially free.

## 2. The VM architecture in concrete terms (§3.1)

```
┌──────────────────────────────────────────────────────────────┐
│  IMAGE FILE (loaded as memory snapshot, Smalltalk/Lisp style)│
│  ┌──────────────────────────────────────────────────────────┐│
│  │  Optimising compiler (written in Factor)                 ││
│  │  Parser (written in Factor — extensible via PARSING:)    ││
│  │  Object system (written in Factor — MOP-extensible)      ││
│  │  Standard library: kernel, sequences, math, strings, ... ││
│  │  [Stock image: also UI, tools, demos — we strip these]   ││
│  └──────────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────────┘
                            ▲ loaded by ▲
┌─────────────────────────────────────────────────────────────┐
│  VM (C++)                                                   │
│   • GC: generational; nursery + aging are copying;          │
│     tenured is mark-sweep-compact.  Code heap is also       │
│     compacted.  Callstack is rewritten on code move. (§3.1) │
│   • Method dispatch: PICs.  Cold → mono/poly (up to 3       │
│     classes) → megamorphic (shared cache). (§3.1)           │
│   • Base compiler: context-threaded, primitive-inlining,    │
│     single pass.  Machine-code templates generated by       │
│     Factor at bootstrap and embedded into the VM. (§3.3)    │
│   • Image loader + saver                                    │
│   • FFI bridge                                              │
│   • Tagged pointers: fixnums inline; everything else        │
│     boxed (compiler often eliminates boxing). (§3.1)        │
└─────────────────────────────────────────────────────────────┘
```

**Key consequence**: the artefacts on disk are
*(a)* the VM executable/DLL (~600 KB), *(b)* an image file (variable size).
The VM by itself is mute; the image carries the parser, object system, and
either the base compiler templates only, or also the optimising compiler.

## 3. The IR pipeline (§3.4)

What the optimising compiler actually does:

```
Factor source / quotation array
    │
    │  (no separate front-end — the stack effect checker IS the front end)
    ▼
─ Stack checker / high-level IR builder ─────────────────────  §3.4.1
    • abstract-interprets the program against stack effects
    • inlines INLINE-declared words
    • inlines quotations passed to row-polymorphic combinators
      (NOT optimisation — semantically required; §2.1.4)
    • expands macros (called at compile time with constant inputs;
      §2.3.2)
    • emits high-level IR: block-structured, SSA
    │
    ▼
─ High-level optimisations ───────────────────────────────────  §3.4.3
    • SCCP with EXTENDED lattice (object types, numeric intervals,
      array lengths, tuple slot types).  Type transfer functions
      can REWRITE nodes inline → cycle of constant-prop + inlining
      + further specialisation until fixed point.
    • Escape analysis + scalar replacement (eliminates tuple
      allocations that don't escape — closures, intermediate
      structs, specialised-array temporaries).
    • Modular-arithmetic overflow-check elision (their novel
      contribution — globally proves `(a + b) & mask` doesn't
      need overflow tracking on the intermediate `+`).
    │
    ▼
─ Construct low-level IR ─────────────────────────────────────  §3.4.4
    • Lowers block structure to CFG of basic blocks
    • Makes stack reads/writes explicit
    • Branch splitting
    • Reconstructs SSA using Briggs/Cooper/Harvey/Simpson '98,
      with Das/Ramakrishna TDMSC algorithm for dominance
      frontier computation
    │
    ▼
─ Low-level optimisations ────────────────────────────────────  §3.4.4
    • Local value numbering (with algebraic identities;
      rewrites array accesses to use x86 addressing modes)
    • Global copy propagation
    • Representation selection (boxed vs unboxed; cost-based;
      picks register storage for floats/SIMD when profitable)
    • Instruction scheduling (Sarkar/Serrano/Simons '01;
      register-pressure-aware)
    │
    ▼
─ Out-of-SSA + register allocation ───────────────────────────  §3.4.4
    • Out-of-SSA via Boissinot et al. '09 (eliminates φ nodes
      with simultaneous copy coalescing)
    • Linear-scan RA, second-chance binpacking variant
      (Traub '98, Wimmer '04)
    │
    ▼
Machine code (x86-64 / ARM64)
```

**What this means for us**: we emit *quotations* (arrays of word/literal cells).
The pipeline above is invoked when those quotations get JIT-compiled (either
eagerly by `jit-compile` or lazily on first call).  We never touch the IR.
The seam is **above the stack checker**: we hand the stack checker something
it can statically infer about.

## 4. The GC (§3.1 — terse but informative)

- Generational.  Nursery + aging = copying collector.  Tenured = mark-sweep-compact.
- Citations: Wilson '92 survey (general technique), no Factor-specific paper.
- **Code heap is also compacted.**  Function pointers move.  The GC walks
  the callstack and rewrites return addresses to relocated targets.  This
  is unusual — most JITs pin code.
- Tuples have an array-based slot layout.  Tuple redefinition (changing the
  class layout) triggers a full GC pass that resizes existing instances
  (§3.2).  We probably never trigger this from Forth, but worth knowing.

**Implications for us:**
- We cannot hold a long-lived `*mut Quotation` or `*mut Word` from Rust
  across operations that may GC.  We pass everything through Factor's
  word/quotation cell-tagged handles.
- Write barriers exist (implied by the generational design; not detailed
  in the paper — see `vm/write_barrier.{cpp,hpp}` for source).  We don't
  emit IR that touches them directly; we use the primitives.

## 5. The bootstrap process — **this changes the slim-image task** (§3.1)

The paper describes the bootstrap as a four-step process:

1. **Image-generation tool** runs in an existing Factor instance, produces
   a *bootstrap image* containing the parser, object system, and core
   libraries.  (Crucially: NOT the optimising compiler.)
2. The VM runs against this bootstrap image, loads a minimal set of
   libraries, compiles them with the **base compiler** (the C++ one).
3. The **optimising compiler** is then loaded (it's Factor code at this
   stage, compiled by the base compiler).
4. Base libraries are *recompiled* with the optimising compiler.  Then
   the full dev environment (tools, UI) is loaded and the image is saved.

**What our manifesto should say differently:**

The old manifesto's Stage 3 said "Build a slim VM from `E:\factor-src\`
with `nf_vm_*` C embedding API exported."  But the *VM doesn't need
modification* — only the image does.  The mission is:

> Take the existing factor.dll.  Run the image-generation tool against
> the unmodified Factor source.  Emit a slim image containing: parser,
> object system, kernel, sequences, math, strings, alien, the optimising
> compiler, `alien.remote-control`, and our `forth.all` runtime
> vocabulary.  Exclude: UI, tools, demos, games, web frameworks, GUI dev
> env.  Result: an image that loads in the standard VM, exposes the
> embedding callbacks via `alien.remote-control`, and contains the
> optimising compiler so that our Rust-emitted quotations get JITted at
> full quality.

The deliverable is therefore a **boot script** that runs in an existing
Factor and saves a new (slim) image, not a forked VM source tree.

If we want to be conservative and only ship the base compiler (slower
generated code but smaller image, no optimising-compiler load), that's a
separate variant.  The paper doesn't say how small the bootstrap image
is, but it's whatever the image-generation tool emits before the
optimising compiler is loaded — probably a few MB.

This is **dramatically less work** than the manifesto suggested.  Stage 3
collapses from "fork the VM, add an embedding API, build it" to "write a
Factor bootstrap script and run it once."

## 6. The stack checker as a real constraint for ANS Forth (§2.1.3, §2.1.4)

Quoting the paper directly:

> In Factor, with very few exceptions, words must pop a fixed number of
> inputs off the stack, and push a fixed number of outputs onto the stack.
> Row-polymorphic combinators, described below, and macros, described in
> 2.3.2, are the only two exceptions to this rule.

ANS Forth is much looser.  Standard words with *runtime-determined* stack
effect:

- `EXECUTE` ( i*x xt -- j*x ) — calls a runtime XT, effect unknown
- `CATCH` / `THROW` — abort up the stack, undefined depth on throw
- `BRANCH` / `?BRANCH` / `LIT` — internal compile-time primitives, fine
- `R@` / `R>` / `>R` — return stack manipulation, statically tractable
- `EVALUATE` / `INTERPRET` — feed a string back through the parser

For the dynamic-effect ones, the paper offers two escape hatches (§2.1.4):

1. **`call( -- )` at the call site** — declares the effect, checked at
   runtime.  Allows dynamic dispatch on a quotation while keeping the
   caller's static effect intact.
2. **`with-datastack`** — call a quotation with an explicit datastack
   array; the parent's stack is not affected.  Paper explicitly cites
   this as "useful for implementing a read-eval-print loop."

Our Forth front-end has two routes:

**Route A (strict):**  Do our own ANS stack-effect inference Rust-side, and
emit only quotations that pass Factor's stack checker.  `EXECUTE` becomes
`call( -- )` with a declared effect that we infer or annotate.  `CATCH`
becomes a `with-datastack` block.  Pros: full optimisation pipeline.  Cons:
real engineering — we essentially reimplement Factor's stack checker for
ANS Forth.

**Route B (loose):**  Wrap every Forth user expression in `with-datastack`
at the REPL level.  Inside, anything goes — the stack checker is bypassed
because each line is a separate quotation called against an explicit
datastack.  Pros: minimal Rust-side work, ANS compatibility falls out.
Cons: defeats inlining across REPL-typed lines.  Performance approaches
"base compiler" rather than "optimising compiler" speed.

**Plausible plan**: ship Route B first to get end-to-end working; tighten
to Route A on a per-word basis where the optimisation matters.  ANS-defined
words that take a fixed effect (`+`, `DUP`, `IF`, etc. — i.e. most of them)
go through Route A straight away because their effects are known.  Only the
genuinely-dynamic ones (`EXECUTE`, `CATCH`, `EVALUATE`) need Route B.

## 7. Performance — calibrating expectations (§3.5)

The paper benchmarks Factor (2010 version) against CPython 3.1.2, SBCL
1.0.38, LuaJIT 2.0.0-beta4, and V8 r4752 on 7 programs from the Computer
Language Benchmark Game.

Summary across the 7 benchmarks:
- **Factor matches or beats LuaJIT** on 4 of 7 (binarytrees, nbody, regexdna, spectralnorm)
- **LuaJIT beats Factor** on 3 (fasta, knucleotide, revcomp)
- **SBCL beats Factor** on 4 of 7 (binarytrees, fasta, knucleotide, spectralnorm)
- **Factor beats V8** on 6 of 7 (V8 only wins regexdna by an order of magnitude)
- **Factor beats CPython** on 6 of 7 (regexdna is the lone tie)

So in 2010 Factor was *in the LuaJIT / SBCL league*, sometimes ahead,
sometimes behind, on real benchmarks.  LuaJIT and SBCL are both top-tier
implementations; this is not a toy.  Sixteen years later the relative
positions may have shifted but the architectural ceiling is high.

For ANS Forth on top: we should expect *competitive with gforth and
swiftforth on the ANS core*, with significant headroom on idiomatic
modern Forth code where the optimiser can specialise (typed locals,
inlined `:` words, fixnum loops).  We will NOT match a stripped-down
STC kernel like Sam Falvo's UCC Forth or VFX on micro-benchmarks of
threaded code dispatch — those have no abstraction tax — but on real
programs the inliner closes the gap.

## 9. What the authors flagged as hard, vs. routine

**Hard** (their original contributions, novel work):
- Stack effect inference *combined with inlining* in a single pass (§2.1.4) — they had to invent this combination
- Modular-arithmetic overflow-check elision (§3.4.3) — explicitly called novel
- Closure elimination as constant propagation (§3.4.3, cite [13] = Ehrenberg's student paper)
- The full pipeline working together at production speed

**Routine** (cited and adopted):
- Tagged pointers (Gudeman '93)
- Generational GC (Wilson '92 survey)
- Mark-sweep-compact for tenured (standard)
- PICs (Hölzle/Chambers/Ungar '91)
- SSA construction (Briggs et al. '98)
- TDMSC dominance frontier (Das/Ramakrishna '05)
- Linear-scan with second-chance binpacking (Traub '98, Wimmer '04)
- Out-of-SSA (Boissinot et al. '09)
- Context-threaded base compiler (Berndl et al. '05)
- SCCP (Wegman/Zadeck '91)

**For our purposes:** the routine items are technical debt-free choices.
The hard items are where the project's value lives, but we don't touch
them — we feed the pipeline above them and consume the machine code below
them.

## 10. Open questions the paper does NOT answer

1. **How exactly does `array>quotation` work?**  The primitive-index shows
   it exists.  Does it just tag an existing array as a quotation, or does
   it copy?  Does it trigger JIT compilation eagerly or lazily?  → Read
   `vm/quotations.cpp` and grep for the primitive.

2. **What is the C ABI for a primitive?**  We see `primitive_dup`,
   `primitive_fixnum_add`, etc. exported from `factor.dll`.  Do they take
   `factor_vm*` and read from `vm.ctx`, or a separate stack pointer?  →
   `vm/primitives.cpp` will tell us; one example suffices.

3. **Is there a documented embedding entry point that avoids
   `start_standalone_factor`?**  The `alien.remote-control` vocab has
   `init-remote-control` but we don't yet know its precondition graph
   (does the image need to be loaded first?  the VM initialised but no
   listener?).  → Stage 2 task #1.

4. **What does the image file format actually look like at the byte
   level?**  We need this both for the bootstrap-image task AND to
   understand what `(save-image)` produces.  → `vm/image.cpp` plus a
   hex-dump of the existing image.

5. **What happens to JIT-compiled code on `(save-image)`?**  Is it
   preserved, or does the image saving discard machine code and
   require re-JITting on next load?  Section 3.1 implies the code heap
   IS part of the image ("The image can be saved, and effectively acts
   as a cache for compiled code") — so yes, preserved.  Confirm in
   source.

Read `basis/alien/remote-control/remote-control.factor` first.  Then
`vm/quotations.cpp` and `vm/primitives.cpp`.  Then look at `vm/image.cpp`
to understand image format.  Then we'll know if Stage 3 collapses to "run
a Factor boot script" (as I now believe) or whether the VM needs
modification after all.

The single biggest mental-model update from this paper: **the VM is not the
thing we modify.  The image is.**  Slim image, standard VM, Rust harness
talking to the image via `alien.remote-control` plus a thin layer of
Factor-side glue we write into the image as part of the bootstrap.
