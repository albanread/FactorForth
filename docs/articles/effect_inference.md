# Stack Effect Inference: Bridging Static Safety and Forth's Flexibility

One of the defining features of Forth is the stack-effect comment: `( a b -- c )`. In classical ANS Forth, these comments are exactly that—comments. If you say a word consumes two numbers and returns one, but the body actually consumes one and returns three, the compiler won't stop you. It will just happily compile the code, which will eventually underflow or overflow the stack at runtime.

Modern functional languages and JIT compilers (like Factor, which NewFactor targets as a backend) have strict requirements. Factor performs rigorous static stack-effect checking on every compiled word. If a word's structural effect is unknown or mismatched, Factor refuses to compile it.

Because NewFactor acts as an ANS Forth front-end to Factor's strict backend, it must somehow bridge Forth's loose, permissive nature with Factor's strict, mathematical execution model. The solution is **"Synth-is-Authoritative" Inference**.

## How NewFactor Understands the Stack

When you define a word in NewFactor, the Rust compiler pipeline parses the body of the word and walks it expression by expression applying Abstract Interpretation.

Consider this sequence:
```forth
1 2 +
```
The inferencer evaluates it:
1. `1` pushes an item `( -- a )`
2. `2` pushes an item `( a -- a b )`
3. `+` consumes two items and pushes one `( a b -- c )`
4. Net effect: `( a b -- c )` which evaluates to consuming 0 items and producing 1.

The analyzer goes further, modeling complex compositional algebra for control structures. For example, in an `IF / THEN` block, the analyzer mathematically demands the branch body must have the net effect `( n -- n )`. If `IF` takes an item and branch executes, the merge point must look identical on the stack regardless of whether the branch was taken.

### The Conflict: Synth vs. Declared

What happens when the `( ins -- outs )` declaration written by the programmer disagrees with the mathematical ground-truth inferred by the compiler?

In the first iteration of NewFactor, a mismatch was an instant compile error. But Forth culture is permissive; developers don't want the compiler stopping them from experimenting because they forgot to update a comment.

The answer is a paradigm shift: **The code is the truth; the comment is just documentation.**

When NewFactor compiles a word, it maintains two maps:
1. `user_effects`: What you wrote. Used by the IDE and `HELP` tools.
2. `body_effects`: What the compiler mathematically proved the word does.

When emitting the final Factor backend IR, NewFactor uses a priority decision table:
- If the signature matches the math, it uses your declaration (preserving meaningful names like `( radius -- area )`).
- If the signature **disagrees** with the math, NewFactor silently relies on the proven `body_effects`. It writes out variables like `( -- r0 r1 )` to ensure Factor's strict JIT successfully compiles the word.

### Diagnostics without Blockers

Because the backend IR relies on the compiler's synthetic effect, NewFactor guarantees that what is handed to Factor is mathematically sound. 

But what about the user's invalid declaration? 
Instead of a fatal compiler abort, NewFactor emits a **warning**. When compiling, you will see output like:
```text
warning: `bad` at line 1, column 3: declared ( -- 0 items) but body produces ( -- 2 items )
```

You are gently warned that your comment has drifted and the program might bug out later if other logic relies on your documentation. However, the system still proceeds to build and execute. 

### Why This Matters

This architecture is deeply pragmatic. 
1. **Safety:** Factor's strict JIT is always satisfied, preventing wild runtime crashes or guard page violations caused by mismatched data flows. 
2. **Analysis:** The mathematical stack knowledge generated by effect synthesis passes on to variable narrowing/escape-analysis models (used to safely instantiate Forth variables).
3. **Culture:** It respects ANS Forth's history as an interactive, permissive REPL language. Your code evaluates without blocking guard-rails while you are testing, but gives you compiler-grade intelligence warnings about internal consistency. 

With "Synth-is-Authoritative" inference, NewFactor lets you write ANS Forth with functional programming discipline, but without the functional programming straitjacket.
